LessonVideoQueueService Remotion 视频生成流程分析
整体架构
视频生成采用三层降级策略：Remotion 本地渲染 → 第三方 API → FFmpeg 兜底。


┌─────────────────────────────────────────────────────┐
│                   HTTP API 层                        │
│  POST lessons/:id/teaching-video/tasks  (创建任务)   │
│  GET  lessons/:id/teaching-video         (下载视频)  │
│  GET  lessons/:id/video-status           (查询状态)  │
└──────────────────────┬──────────────────────────────┘
                       │ enqueue()
┌──────────────────────▼──────────────────────────────┐
│          LessonVideoQueueService (队列调度)           │
│  - 定时轮询 pending 任务 (默认 3s)                    │
│  - CAS 原子领取 (status=pending → processing)         │
│  - 缓存复用 (cacheKey 去重)                           │
│  - 自动重试 (最多 2 次)                               │
│  - 过期清理 (默认 72h)                                │
└──────────────────────┬──────────────────────────────┘
                       │ generateVideoBuffer()
         ┌─────────────┼─────────────────┐
         ▼             ▼                 ▼
  ┌────────────┐ ┌────────────┐  ┌──────────────┐
  │  Remotion   │ │ 第三方 API  │  │ FFmpeg 兜底   │
  │ (本地渲染)  │ │ (远端生成)  │  │ (AiService)  │
  └────────────┘ └────────────┘  └──────────────┘
核心流程详解
1. 任务入队 — enqueue() (lesson-video-queue.service.ts:108)
从 Content 实体中提取课程数据，调用 buildPackPayloadFromContent() 构建结构化 payload
payload 包含：title, topic, summary, ageGroup, watchScene, visualStory, videoLesson, modules(listening/reading/writing/game/quiz)
用 sha1(contentId + updatedAt + payload) 计算 cacheKey 做去重
如果存在相同 cacheKey 且未过期的已完成任务，直接复用
2. 队列消费 — processQueue() (lesson-video-queue.service.ts:161)
onModuleInit 启动定时器，每 3s 轮询一次
单 worker 模式 (workerBusy 标志位防止并发)
每 120 个 tick (约 6 分钟) 清理一次过期缓存
3. 任务领取 — processNextTask() (lesson-video-queue.service.ts:178)
查找最旧的 pending 任务
使用 SQL UPDATE ... WHERE status='pending' 做 CAS 原子领取
失败自动重入队列（attemptCount 递增，超过 maxRetries=2 标记 failed）
4. 三层降级生成 — generateVideoBuffer() (lesson-video-queue.service.ts:234)

Remotion (本地) → 失败 → 第三方 API (远端) → 失败 → FFmpeg (AiService)
层级 1: Remotion 本地渲染 — generateByRemotion() (lesson-video-queue.service.ts:263)
这是重点路径，调用 RemotionRenderService：

Step 1: resolveComposition() — 决定用哪个 Remotion composition

normalizePayload() — 将输入标准化为 LessonVideoPayload
hasLessonVideoSource() — 检查是否有结构化课程数据（watchScene/visualStory/videoLesson/modules）
如果有 → 走 buildVideoDataFromLesson() 路径（从课程内容构建幻灯片）
如果是纯数字主题 → 返回 NumbersVideo composition
否则 → 调用 GenerateVideoDataTool（AI 工具）生成幻灯片数据
Step 2: buildVideoDataFromLesson() — 幻灯片构建流水线


watchScene/visualStory/videoLesson
         │
         ▼
  buildWatchSlides()         ← 从场景数据构建观看幻灯片（最多8张）
         │                        使用 deriveWatchSceneDocument() 兜底
         ▼
  mergeListeningIntoWatchSlides() ← 将听力脚本合并进观看幻灯片的 narration
         │
         ▼
  buildSupplementSlides()    ← 构建补充幻灯片：
         │                      - reading (读一读)
         │                      - writing (写一写)
         │                      - game (练一练)
         │                      - quiz (评一评)
         ▼
  合并并裁剪到最多 8 张幻灯片
每张幻灯片通过 resolveSceneTheme() 根据模板 ID 匹配预设主题：

science.seasons-cycle — 四季轮转
science.day-night-cycle — 白天黑夜
science.water-cycle — 水循环
science.plant-growth — 植物生长
language.character-stroke — 汉字笔顺
language.word-reveal — 词语揭示
language.story-scene — 故事场景
Step 3: generateNarrationAudioFiles() — TTS 语音合成

逐张幻灯片串行调用 VoiceService.textToSpeech()（避免 TLS 连接池耗尽）
每次最多重试 2 次，间隔 1s
生成的 MP3 文件写入 video-remotion/public/ 目录，文件名为 narration-{sha1}.mp3
解析 MP3 帧头计算精确时长（帧数），失败则按 4 字符/秒估算
每张幻灯片最少 6 秒，加上 0.3s 入场 + 0.5s 尾部缓冲
Step 4: renderComposition() — 调用 Remotion CLI 渲染

将 inputProps 写入临时 JSON 文件
通过 child_process.spawn('npx', ['remotion', 'render', ...]) 调用 Remotion
并发度 = cpuCount - 1（最多 4）
编码格式 h264
解析 stdout/stderr 的百分比进度并更新数据库
渲染完成后清理临时文件和叙述 MP3
层级 2: 第三方 API — generateByThirdParty() (lesson-video-queue.service.ts:297)
POST 创建任务 → 轮询状态（最多 120 次 × 2.5s = 5 分钟）
完成后下载视频到本地
带完整的安全校验：协议限制、主机白名单、私有 IP 拦截、文件大小限制
层级 3: FFmpeg 兜底
调用 AiService.renderTeachingVideoFromPack(payload) 生成简单视频
5. 视频交付
生成的 MP4 写入 storage/lesson-videos/{cacheKey}.mp4
下载接口支持 HTTP Range 请求（视频拖拽/seek），见 learning.controller.ts:468
审批机制：生成后 approvalStatus=pending_approval，家长审批后学生才能观看
关键配置参数
环境变量	默认值	说明
VIDEO_TASK_POLL_INTERVAL_MS	3000	队列轮询间隔
VIDEO_CACHE_TTL_HOURS	72	视频缓存有效期
VIDEO_TASK_MAX_RETRIES	2	最大重试次数
VIDEO_PROVIDER_BASE_URL	-	第三方 API 地址
VIDEO_PROVIDER_MODE	hybrid	hybrid/local_only/off

    ▼

    │   每个场景 → resolveSceneTheme() 匹配模板主题
    │   每个场景 → buildSlideFromScene() 构建幻灯片
    │
    ├─ modules.listening → mergeListeningIntoWatchSlides()
    │
    ├─ modules.reading/writing/game/quiz → buildSupplementSlides()
    │
    ▼
TeachingVideoData (title + subtitle + slides[])
    │
    ├─ 逐张幻灯片 TTS → generateNarrationAudioFiles()

    │
    ▼

    │
    ├─ npx remotion render → MP4 文件
    │
    ▼

remotion 生成视频，视频生成质量由哪些部分决定
基于之前阅读的代码，Remotion 视频生成质量由以下几个层面决定：

1. 数据输入层 — 内容丰富度
核心在 buildPackPayloadFromContent() 从 Content 实体提取课程数据：


Content.content (JSON) → steps[].module → payload
质量取决于课程 JSON 中这六个模块的完整程度：

模块	对应幻灯片	数据要求
watch.scene	观看幻灯片（主内容）	需有 scenes[]，每个 scene 含 visual, narration, onScreenText
listen.listening	合并到观看幻灯片	audioScript[] 提供 narration 增强
read.reading	读一读幻灯片	keywords[], questions[], text
write.writing	写一写幻灯片	tracingItems[], practiceTasks[]
practice.game	练一练幻灯片	activityType, activityData.questions/pairs
assess.quiz	评一评幻灯片	questions[].question
如果课程数据稀疏，代码会在多处兜底：

deriveWatchSceneDocument() 用正则从文本推断背景类型、模板类型
buildFallbackSlide() 生成最简单的单张幻灯片
narration 缺失时使用通用模板文案如 "请和老师一起学习。"
结论：课程 JSON 中 scene 数据越完整（含 characters、items、timeline、templateId），视频内容越丰富。

2. 场景模板层 — 视觉多样性
resolveSceneTheme() 决定每张幻灯片的视觉风格：

支持的模板（质量从高到低）：

模板 ID	视觉效果	触发条件
science.seasons-cycle	四季轮转动画	课程数据显式指定
science.day-night-cycle	昼夜变化	课程数据显式指定
science.water-cycle	水循环动画	课程数据显式指定
science.plant-growth	植物生长	课程数据显式指定
language.character-stroke	汉字笔顺	自动检测"认识X字"
language.word-reveal	词语逐个出现	自动提取中文词语
language.story-scene	故事场景	兜底模板
无模板	按 index 循环调色板	默认
关键问题：模板主要由 scene.visual.templateId 显式指定或由 inferWatchTemplateHint() 通过正则推断。推断的准确度直接影响视觉质量。

颜色方面，10 种背景色 + 10 种强调色按 index 循环分配（BG_PALETTE），如果场景数超过 10 会出现重复配色。

3. TTS 语音层 — 叙述质量
generateNarrationAudioFiles() 是视频时长和节奏的基础：

质量影响因素：

narration 文案质量 — 直接决定语音内容。每张幻灯片 narration 限制 100 字符（slice(0, 100)），可能截断
TTS 服务稳定性 — 串行调用避免 TLS 耗尽，最多重试 2 次
时长精度 — parseMp3DurationFrames() 解析 MP3 帧头计算精确时长：
成功：实际秒数 × 30fps + 0.3s入场 + 0.5s尾部
失败：按 字数 / 4字符每秒 估算
最少 6 秒
如果 TTS 完全失败，幻灯片没有音频但有估算的 durationFrames，视频仍会生成但为无声。

4. 幻灯片数量与裁剪策略
buildVideoDataFromLesson() 中的裁剪逻辑：


const reservedSupportCount = Math.min(4, supportSlides.length);
const maxWatchSlides = Math.max(1, 8 - reservedSupportCount);
const slides = [
  ...mergedSlides.slice(0, maxWatchSlides),
  ...supportSlides,
].slice(0, 8);  // 硬上限 8 张
总量上限 8 张幻灯片
补充幻灯片（读/写/练/评）预留最多 4 张位置
观看幻灯片被压缩到 8 - 补充数量
如果课程只有 watch 模块没有其他模块，可以有最多 8 张内容幻灯片；如果六个模块都有，观看内容会被大幅裁剪
5. Remotion 渲染层
runRemotionRender() 的技术参数：

参数	当前值	影响
编码	h264	兼容性好，质量中等
并发	cpuCount - 1（最多4）	影响渲染速度而非质量
帧率	30fps（由 Remotion 项目定义）	流畅度
分辨率	由 Remotion composition 定义（代码中未指定）	清晰度
compositionId	TopicVideo 或 NumbersVideo	Remotion 端的组件实现
视频的最终渲染效果（动画、布局、字体）取决于 video-remotion/ 目录中 Remotion 项目的 React 组件实现，这部分代码在 video-remotion 子目录中，当前的 NestJS 代码只负责传入 inputProps。

6. 数据丢失点总结
质量可能在以下环节降级：

降级点	位置	原因
字符截断	title: slice(0, 12), subtitle: slice(0, 20)	标题/副标题限制过短，中文6字标题
narration 截断	slice(0, 100)	叙述文案被截断
items 限制	slice(0, 4)	每张幻灯片最多 4 个条目
场景限制	slice(0, 8)	最多 8 个场景
模板推断失败	inferWatchTemplateHint()	正则匹配不到时退化为 story-scene 模板
TTS 失败	串行调用	语音缺失导致无声视频
兜底文案	多处 toText(x, '请和老师一起学习。')	数据缺失时用通用文案
总结：视频质量的决定权重排序为：

课程 JSON 数据质量（场景、视觉、叙述的完整度）— 权重最大
Remotion 组件实现（动画、布局、渲染效果）
模板匹配准确度（templateId 推断）
TTS 语音质量
渲染参数（编码、分辨率）
